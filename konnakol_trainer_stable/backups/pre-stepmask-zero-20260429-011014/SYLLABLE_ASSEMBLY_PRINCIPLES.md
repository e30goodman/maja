# Принципы сборки силлабов в `konnakol_trainer_stable`

Документ фиксирует, как именно в проекте формируются слоги (konnakol syllables) для каждой клетки сетки.

## 1) Источник истины

Основная логика находится в `src/sequencerLabels.ts`:

- словарь слогов: `KONNAKOL_DICTIONARY`
- вычисление плотности: `computeNps(bpm, phraseLen)`
- выбор Kalam (slow/medium/fast): `pickKalam(...)`
- сборка длинных сегментов: `composeLongBar(...)`
- финальная раскладка по клеткам: `buildRowCellSyllableLabels(...)`

`src/SequencerGrid.tsx` только потребляет готовый результат и рендерит `rowCellLabels[cIdx]?.[i]`.

## 2) Базовый словарь (Gati x Kalam): инженерная реализация и канон

Словарь задан для Gati `1..9`, отдельно для `slow | medium | fast`.
Ниже зафиксированы две плоскости:

- **Текущее поведение движка** (что реально отдается из `KONNAKOL_DICTIONARY`);
- **Каноничные карнатические формулы** (что ожидает практик коннакола на слух).

- Gati=1: `Ta`
- Gati=2: `Ta Ka`
- Gati=3: `Ta Ki Ta`
- Gati=4: slow/medium `Ta Ka Dhi Mi`, fast `Ta Ka Ju Nu`
- Gati=5: `Ta Ka Ta Ki Ta`
- Gati=6: slow/medium `Ta Ka Dhi Mi Ta Ka`, fast `Ta Ka Ju Nu Ta Ka`
- Gati=7: slow/medium `Ta Ka Dhi Mi Ta Ki Ta`, fast `Ta Ka Ju Nu Ta Ki Ta`
- Gati=8: slow `Ta Ka Dhi Mi Ta Ka Dhi Mi`, medium/fast `Ta Ka Dhi Mi Ta Ka Ju Nu`
- Gati=9: slow/medium `Ta Ka Dhi Mi Ta Ka Ta Ki Ta`, fast `Ta Ka Ju Nu Ta Ka Ta Ki Ta`

Важный принцип: в быстрых режимах для части паттернов используется переход `Dhi Mi -> Ju Nu`.

### Каноничные соответствия (рекомендуемая музыкальная нормализация)

Ниже формы, которые обычно считаются более аутентичными для обучения карнатической ритмике:

- Gati=5 (Khanda): **предпочтительно** `Ta Di Gi Na Thom` (вариант `Ta Dhi Gi Na Thom`) вместо нейтрального `Ta Ka Ta Ki Ta`;
- Gati=6: чаще группировка `3+3` -> `Ta Ki Ta Ta Ki Ta`; также используется `Ta Di Gi Na Thom Ka` (`5+1`);
- Gati=7 (Misra): характерная пульсация `3+4` -> `Ta Ki Ta Ta Ka Dhi Mi` (а не только `4+3`);
- Gati=9 (Sankirna): канонично как `4+5` или `5+4`, например `Ta Ka Dhi Mi Ta Di Gi Na Thom`.

Инженерно словарь может оставаться математически валидным, но для музыкальной аутентичности стоит хранить отдельный "canonical/teaching" слой фраз.

## 3) Как выбирается Kalam (текущий UX-алгоритм)

Сначала считается `NPS = BPM * phraseLen / 60`, где:

- `phraseLen = subdivs`, если клетка имеет локальные поддоли (`subdivs > 1`);
- `phraseLen = segLen`, если это сегмент соседних клеток с `subdivs == 1`.

Далее `pickKalam` выбирает `slow/medium/fast`:

- стартово (без предыдущего состояния): `<=4.0 slow`, `<=8.0 medium`, иначе `fast`;
- при последующих расчётах используется гистерезис:
  - `slow -> medium` только выше `4.4`
  - `medium -> slow` только ниже `3.6`
  - `medium -> fast` только выше `8.4`
  - `fast -> medium` только ниже `7.6`

Это сделано, чтобы подписи не "дёргались" при мелких колебаниях BPM.

### Музыкальный комментарий (канон Kalam)

В карнатической традиции Kalam трактуется как дискретная прогрессия (Anuloma), а не как непрерывная шкала:

- 1-й Kalam (Vilamba) -> база;
- 2-й Kalam (Madhya) -> ровно `x2` от базы;
- 3-й Kalam (Druta) -> ровно `x4` от базы.

Следовательно, NPS+гистерезис в движке — осознанный UX-компромисс для произвольных BPM, а не буквальная имитация канонического перехода по Kalam.

## 4) Нормализация входа перед сборкой

В `buildRowCellSyllableLabels(...)`:

- каждое `subdivs` приводится в диапазон `1..9`;
- если значение невалидно, берётся `1`;
- `deadStart` ограничивается диапазоном `0..rowSyllCount`.

Инвариант: результат всегда массив длины `rowSyllCount`.

## 5) Алгоритм прохода по ряду

Проход выполняется слева направо:

1. Если клетка попадает в dead tail (`cIdx >= deadStart`) -> в результат пишется `[]`.
2. Если `subdivs > 1` -> это отдельная локальная фраза:
   - выбирается Kalam по `NPS(bpm, subdivs)`;
   - в клетку пишется весь массив слогов длины `subdivs`.
3. Если `subdivs == 1` -> собирается непрерывный сегмент соседних `1`:
   - длина сегмента `segLen`;
   - Kalam по `NPS(bpm, segLen)`;
   - берётся фраза для `segLen`;
   - каждый шаг сегмента получает по 1 слогу (`[phrase[i]]`).

Если в фразе вдруг нет индекса (защитный случай), используется fallback `Ta`.

### Важная карнатическая интерпретация `subdivs > 1`

Если внутри клетки `subdivs > 1`, это фактически локальная **Nadai/Gati внутри Akshara**:

- длительность клетки (Akshara) сохраняется;
- меняется внутренняя плотность слогов (микротайминг);
- первый слог локальной группы должен восприниматься как микро-опора пульса.

При цепочке разных плотностей (`4 -> 5 -> 3` и т.п.) возникает **Nadai Bhedam**. Для музыкальной читаемости важно маркировать старт каждой клетки (акцентом/визуальным маркером), иначе теряется базовый пульс Akshara.

## 6) Сегменты длиннее 9: текущее состояние и рекомендуемый апгрейд

`composeLongBar(segLen, kalam)` работает жадно блоками до 9:

- берётся `9`, затем остаток (`9 + 9 + ... + remainder`);
- для каждого блока используется тот же `kalam`;
- итоговая длина всегда равна `segLen`.

Пример: `segLen=12` -> `Dict[9] + Dict[3]`.

С инженерной точки зрения это корректно по длине, но музыкально часто несимметрично.
Для карнатической фразировки более естественна **Sarva Laghu симметрия**.

Рекомендуемая стратегия при сборке длинных сегментов:

1. сначала искать равномерную раскладку (делители `4`, затем `3`, затем `5`, затем `2`);
2. если делимость есть — собирать повтором равных блоков (`12 -> 4+4+4` или `3+3+3+3`, `10 -> 5+5`, `6 -> 3+3`);
3. жадный остаточный режим применять как fallback для простых/неудобных длин (`11 -> 4+4+3`).

Итог: приоритет музыкальной симметрии над "максимальным первым блоком".

## 7) Память состояния Kalam (sticky behavior)

Опциональная `kalamMap` хранит предыдущее состояние по ключам:

- для локальных ячеек: `${rowIdx}-c${cellIdx}`
- для сегментов `subdivs==1`: `${rowIdx}-seg${segStart}`

За счёт этого гистерезис применяется стабильно между перерисовками.

В `SequencerGrid.tsx` используется `useStableRowCellLabelsCache(...)`:

- передаёт `kalamMap` в `buildRowCellSyllableLabels`;
- собирает `touchedKeys`;
- удаляет stale-ключи (GC), которые больше не существуют в текущей форме сетки.

## 8) Практические следствия

- Слоги зависят не только от выбранного Gati, но и от BPM через Kalam.
- Две одинаковые по размеру клетки могут иметь разные слоги в разный момент при смене темпа.
- Dead cells никогда не получают текста.
- Сетка поддерживает как "локальные фразы в клетке", так и "протяжённый Sarva Laghu сегмент" по ряду.
- Для музыкальной аутентичности желательно разделять "инженерный словарь" и "каноничный обучающий словарь".
- При переменной плотности между соседними клетками критично маркировать старт Akshara (Nadai Bhedam читается по опоре первой доли).

## 9) Чем подтверждено (тесты)

Покрытие находится в `src/sequencerLabels.test.ts` и проверяет:

- формулу NPS;
- пороги и гистерезис Kalam;
- корректность словаря для Gati `1..9`;
- переход `Dhi Mi -> Ju Nu` в fast-паттернах;
- сборку длинных сегментов (`composeLongBar`);
- корректную работу dead хвоста;
- смешанные сценарии (`subdivs>1` + сегменты из `1`);
- sticky-поведение с `kalamMap` и контракт `touchedKeys`.

Для каноничной карнатической линии дополнительно полезны тесты:

- каноничные формы для `5/6/7/9` (отдельный профиль словаря);
- порядок приоритета симметричной сборки длинных сегментов;
- инвариант маркировки первой доли Akshara при `subdivs`-сменах.

## 10) Реализация пауз (Karvai) через битовые маски

Документируем внедрение пауз (тихих шагов) внутри существующей сетки без добавления новых UI-меню.
Каркас: пауза реализуется через mute отдельных шагов внутри `subdivs`.

### 10.1) Новая структура данных ячейки (битовая маска)

Вместо модели "только число `subdivs`" ячейка поддерживает маску шагов, например `stepMask: boolean[]`.

- длина маски = локальный `subdivs` (локальный Gati);
- `true` = шаг звучит (удар);
- `false` = шаг заглушен (Karvai/пауза).

Пример: `stepMask = [true, false, true, false]` означает `subdivs = 4`, где звучат только шаги `0` и `2`.

Для обратной совместимости старый формат (одно число `subdivs`) неявно нормализуется в "все звучит":

- `subdivs = N` -> `stepMask = [true, true, ...]` длиной `N`.

### 10.2) Инвариант NPS (Notes Per Second) не меняется

Критическое правило: в коннаколе пауза — полноценный шаг фразы, произносимый "внутри" (про себя), а не удаленный из метра.

Поэтому при вычислении:

- `NPS = BPM * phraseLen / 60`

величина `phraseLen` всегда равна **полной длине маски** (`stepMask.length`), независимо от количества `false`.

Следствия:

- `pickKalam(...)` работает по прежним порогам/гистерезису;
- Nadai-сетка не "схлопывается" из-за тишины;
- наличие пауз не должно менять выбор `slow/medium/fast` по сравнению с "полностью звучащей" фразой той же длины.

### 10.3) Наложение маски на словарь (мутация слогов)

Для локальной клетки (`subdivs > 1`) процесс такой:

1. берется `L = stepMask.length`;
2. из `KONNAKOL_DICTIONARY` извлекается базовая фраза длины `L` для выбранного Kalam;
3. фраза маппится через `stepMask`:
   - `stepMask[i] === true` -> оставить исходный слог;
   - `stepMask[i] === false` -> заменить на символ паузы (`"-"` или `","` по принятому в UI соглашению).

Пример:

- Gati `4`, `stepMask = [true, false, true, true]`
- база: `['Ta', 'Ka', 'Dhi', 'Mi']`
- результат: `['Ta', '-', 'Dhi', 'Mi']`

### 10.4) Разрыв сегментов в `composeLongBar(...)` для `subdivs == 1` (Sarva Laghu)

Для цепочек клеток с `subdivs == 1` пауза должна разрывать непрерывный сегмент чтения.

Если текущая клетка имеет `stepMask = [false]`:

- накопленный до нее сегмент `[true]`-клеток закрывается;
- закрытый сегмент считает `segLen`, получает Kalam через обычный NPS и собирается через `composeLongBar(...)`;
- сама паузная клетка получает `"-"`;
- после паузы стартует новый сегмент.

Алгоритм:

1. накапливать подряд клетки со `stepMask = [true]`;
2. при встрече `stepMask = [false]` — завершить сегмент и отрендерить паузу;
3. продолжить накопление с следующей звучащей клетки.

Пример ряда:

- вход: `[true, true, true, false, true, true, true]`
- разбиение: `Сегмент(3) + Пауза + Сегмент(3)`
- рендер: `['Ta', 'Ki', 'Ta', '-', 'Ta', 'Ki', 'Ta']` (а не единый паттерн длины `7`).

### 10.5) Кэш и sticky behavior (`kalamMap`)

Для сегментов `subdivs == 1` ключи вида `${rowIdx}-seg${segStart}` остаются корректными, но `segStart` теперь обязан пересчитываться после каждого разрыва паузой.

Это гарантирует:

- независимую sticky-историю для сегментов по обе стороны Karvai;
- отсутствие "перетекания" предыдущего Kalam через паузную границу.

Локальные ключи ячеек `${rowIdx}-c${cellIdx}` остаются без изменения, потому что мутация (`слог -> "-"`) происходит внутри уже выбранной локальной фразы клетки.

## 11) Архитектурный план внедрения "0" (правильный путь через `stepMask`)

Ниже пошаговый план внедрения `0` как паузы без ломки контракта `subdivs`.
Ключевая идея: `subdivs` остается плотностью (`>=1`), а "0" в UI — это представление состояния маски (`stepMask`), где отключен нужный шаг.

### 11.1) Контракты и инварианты (фиксируем до кода)

1. `subdivs` не хранит `0` в данных домена и в snapshot wire-format.
2. Пауза хранится только в `stepMask`.
3. `phraseLen` для `computeNps` равен длине маски, а не числу активных шагов.
4. `dead tail` обрабатывается раньше масок и не меняется.
5. UI никогда не подставляет числа (`subdivs`, `i`, `length`) в текст слога при пустом шаге.

### 11.2) Data model (App state + snapshot)

Добавить поле:

- `cellStepMasks: Record<string, boolean[]>` (ключ `${row}-${cell}`).

Правила нормализации:

- если маски нет -> "все шаги активны";
- если длина маски не равна `subdivs` -> расширить/обрезать до `subdivs` (дефолт `true`);
- если после toggle маска полностью `true`, можно удалять ключ для компактности.

Snapshot:

- включить `cellStepMasks` в `createEmptySnapshot`, `snapshotToJSON`, `parseSnapshotRow`;
- сохранять и восстанавливать без влияния на старые снапшоты (обратная совместимость: отсутствие поля = все `true`).

### 11.3) Ввод пользователя (два режима Divs)

Обычный режим (long-press/drag по клетке, `SequencerGrid.tsx`):

- оставить текущую механику выбора `subdivs` без изменения UX;
- добавить действие mute по конкретному подшагу: `toggleCellStepMute(cellKey, stepIdx, subdivs)`;
- операция меняет только `stepMask`, не `subdivs`.

Режим распущенного слайдера (`App.tsx`, Divs-panel):

- слайдер продолжает управлять только `subdivs`;
- "0" в UI показывается как состояние паузы (например, если выключен отображаемый шаг), но не как реальное значение `subdivs` в домене;
- при изменении `subdivs` маска для клетки ре-нормализуется до новой длины.

### 11.4) Label engine (`sequencerLabels.ts`)

Расширить `buildRowCellSyllableLabels(...)` опцией:

- `cellStepMasks?: Record<string, boolean[]>`.

Локальные клетки (`subdivs > 1`):

1. взять `phraseLen = stepMask.length`;
2. выбрать Kalam через `computeNps(bpm, phraseLen)`;
3. построить базовую фразу длины `phraseLen`;
4. применить маску: `finalSyl = stepMask[i] ? phrase[i] : '-'`.

Сегменты `subdivs == 1`:

- если `stepMask[0] === false`, это паузная клетка:
  - закрыть текущий Sarva Laghu сегмент,
  - положить `['-']`,
  - начать новый сегмент после паузы.

Это устраняет ложное "протекание" длинного сегмента через паузу.

### 11.5) Cache invalidation / sticky Kalam

`useStableRowCellLabelsCache` должен зависеть не только от `customSubdivisions`, но и от сигнатуры масок.

Добавить:

- `stepMaskSigByRow` (или эквивалент), включенный в `useMemo` deps;
- в `kalamMap` ключах учитывать сигнатуру маски (например, `-m<maskSig>`), чтобы hysteresis не смешивал состояния с разными паузами.

### 11.6) Render contract (`SequencerGrid.tsx`)

Строгое правило рендера текста шага:

- читать только `rowCellLabels[cIdx]?.[i]?.syl ?? ''`;
- если `'-'/'–'/''` -> рендерить пустоту;
- без fallback на числа.

Это гарантирует, что пауза не превращается в "9"/индекс/длину из-за JS truthy/falsy fallback.

### 11.7) Audio/MIDI parity

Audio (`App.tsx` scheduler) и MIDI (`midiExport.ts`) должны применять ту же маску:

- muted-step не эмитит звук/ноту;
- тайминг сетки не сжимается;
- first-beat/Ta/accent правила работают поверх маски, но в рамках действующих TA-инвариантов.

### 11.8) Порядок внедрения (execution order)

1. Data model + snapshot roundtrip (`cellStepMasks`).
2. UI toggle mute-step (без изменения движка слогов).
3. `buildRowCellSyllableLabels` с масками.
4. Cache invalidation + `kalamMap` mask-aware keys.
5. Render contract без fallback-чисел.
6. Audio/MIDI parity.
7. Cleanup: удалить временные маппинги, если были.

### 11.9) Тест-план (обязательный минимум)

Unit (`sequencerLabels.test.ts`):

- `phraseLen = stepMask.length`, не `count(true)`;
- локальная маска для `subdivs=4` -> `['Ta','-','Dhi','Mi']` (примерно);
- разрыв сегмента на `subdivs==1` при `stepMask[0]=false`;
- dead-tail приоритет над маской.

Integration:

- клик mute шага -> пересчет `rowCellLabels` в том же кадре;
- пауза рендерится пусто;
- смена `subdivs` нормализует маску длины.

Snapshot:

- save/load сохраняет `cellStepMasks`;
- старый snapshot без `cellStepMasks` загружается корректно (все шаги активны).

Audio/MIDI:

- muted-step не звучит в реальном времени;
- muted-step отсутствует в MIDI;
- темп/Kalam не "прыгают" из-за паузы.

### 11.10) Критерий готовности

Фича считается завершенной, когда одновременно верны условия:

- `Divs` выбор не сломан в обоих режимах;
- пауза визуально пустая, без цифр;
- snapshot roundtrip полностью сохраняет маски;
- audio/MIDI/GUI дают одинаковый результат для пауз;
- нет регрессий по TA/first-beat/dead-tail/poly lane separation.
