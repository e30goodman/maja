# AI Deep Audit Guide

## Манифест проблемы: ловушка Symptom Patching

Когда UI показывает "залипание" или "частичное обновление", ИИ часто застревает в самом видимом слое: React-компоненты, `useMemo`, `React.memo`, `useEffect`, `useRef`.
Это когнитивная ловушка: чинится симптом рендера, а не источник данных.

Результат ловушки:
- растет количество локальных костылей в UI-слое;
- корневая ошибка в генераторе данных продолжает жить;
- баг выглядит "рандомным", хотя он детерминирован и воспроизводим;
- каждая новая "фиксация рендера" отравляет контекст и уводит дальше от причины.

Главный принцип: **если путь данных сломан upstream, React не обязан спасать это downstream**.

## Триггер активации: когда ИИ обязан остановиться

При любом из условий ниже ИИ обязан прекратить писать новый UI-код и перейти в режим аудита:

1. Две подряд попытки чинить state/render не меняют поведение UI по факту.
2. `Index 0` работает корректно, а `Index 1+` стабильно ломается.
3. Появляется импульс "форснуть ререндер" через `useRef`, `useEffect`, счетчики версий, watchdog-паттерны, дублирование JSX.
4. Есть асимметрия между соседними элементами списка при одинаковом lifecycle (обычно признак различий входных данных, а не React-механики).
5. Для объяснения бага приходится вводить "магические" гипотезы про мемоизацию без подтвержденного A/B-трейса входных данных.

Обязательное правило остановки:
**No new patch before trace proof.**

## Протокол Дифференциального Аудита (A/B Trace)

### Шаг 0. Зафиксировать симптомы как детерминированный кейс

- Сформировать минимальный воспроизводимый сценарий.
- Отдельно записать "здоровый" элемент (`A`: обычно `Index 0`) и "больной" (`B`: обычно `Index 1+`).
- Вести сравнение только на одном и том же snapshot входа.

### Шаг 1. Доказать корректность ключа входа (`cellKey`)

- Логировать/проверить фактический `cellKey` для `A` и `B` в точке входа в вычисление.
- Убедиться, что `cellKey` стабилен и соответствует ожидаемому индексу/доменной сущности.
- Исключить коллизии/смещения ключей до начала любых гипотез о рендере.

Если этот шаг не доказан, **дальше идти нельзя**.

### Шаг 2. Сравнить полный маршрут A vs B

Построить "трассу данных" по одинаковым контрольным точкам:

1. Входные структуры до генерации.
2. Промежуточные сегменты внутри циклов/группировок.
3. Результат pure-генератора.
4. Значение на входе в React-компонент.
5. Значение внутри memo-слоя (если есть).
6. Финальный проп, который отображается в UI.

На каждом этапе фиксировать различия между `A` и `B`.
Первая точка расхождения = кандидат в корневую причину.

### Шаг 3. Приоритет проверки слоев (обязательный порядок)

1. **Upstream Generators** (чистые функции, сборщики данных, трансформеры).
2. Контракты вход/выход между слоями.
3. Только после этого — React cache layer (`useMemo`, `React.memo`, dependency arrays).

Запрещено перепрыгивать сразу к React-механике, если нет доказательства, что upstream-данные эквивалентны и корректны.

### Шаг 4. Правило доказательства перед фиксом

Перед любым код-фиксом ИИ обязан ответить на два вопроса:

- Где первая точка расхождения между `A` и `B`?
- Это расхождение рождается до React-слоя или внутри React-слоя?

Если ответа нет, фикс запрещен.

## Технический урок: Segment Loop Blindness

Наш инцидент: функция `buildRowCellSyllableLabels` внутри цикла пакетной сборки сегментов проверяла условия на уровне группы, но пропускала индивидуальные маски элемента.
Итог: `Cell 0` случайно проходил "здоровый" путь, а `Cell 1+` получали залипшие/некорректные labels.

Урок:
- в batch/segment loops всегда валидировать **пер-элементный контракт**, не только агрегатный;
- "один элемент работает" не доказывает корректность алгоритма;
- индексная асимметрия почти всегда сигналит о data-shape divergence внутри генератора.

## Анти-костыли (запрещено до завершения A/B-аудита)

- Добавлять `useRef`/`useEffect` для ручного форса обновления.
- Дублировать JSX-ветки "для надежности".
- Вводить watchdog/version-counter без доказанной причины.
- Переписывать memo-зависимости "наугад".
- Обвинять React до фиксации upstream-трассы.

## Мини-чеклист перед merge

- Доказана корректность `cellKey` для `A` и `B`.
- Найдена первая точка расхождения в A/B-трейсе.
- Проверен upstream generator path до UI.
- Фикс поставлен в слое первопричины, а не в слое симптома.
- UI-костыли не добавлены.
